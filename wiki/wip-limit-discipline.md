# WIP=1 discipline

**Summary** : Work-in-Progress limit fixé à 1 pour les agents. Finir une tâche complètement avant d'en commencer une autre. Anthropic mesure +37% de taux de complétion vs prompts larges. Le nombre de lignes de code écrites est **négativement** corrélé à la complétion : plus d'output ≠ plus de features finies.

**Sources** : `raw/ingested/lecture-07-draw-clear-task-boundaries-for-agents.md`

**Last updated** : 2026-05-24

---

## Contenu

### Le principe

> "WIP=1 is the default safe setting for agent harnesses — finish one, then start the next; don't try to parallelize." — Lecture 07

Concept emprunté à Kanban (David Anderson). Pour les agents, **WIP=1** = une seule tâche `active` à un instant donné. Les autres sont `not_started` ou `blocked`.

### Pourquoi pas WIP=N pour les agents

Hypothèse intuitive : "un agent peut faire 3 trucs en parallèle". Réalité observée :

| WIP | Output (lignes de code) | Verified completion rate |
|-----|-------------------------|--------------------------|
| 5 | 800 | 20% |
| 1 | 200 | 100% |

**Moins de code, plus de features finies.** Anti-intuitif mais documenté.

### Mathématiques (attention budget)

Si C = capacité totale de contexte/attention et k = nombre de tâches actives :

```
attention par tâche = C / k
```

Quand C/k < threshold minimum pour qu'une tâche soit faite proprement → **aucune** ne finit. C'est mathématique, pas métaphorique.

### Loi de Little

`L = λ × W` (lead time = throughput × WIP).

Conséquence : multiplier WIP ne multiplie pas le throughput — ça multiplie le lead time par tâche. Plus chaque tâche traîne, plus la probabilité d'échec en cours de route augmente.

### L'overreach-under-finish symbiosis

Vicious cycle observé :

```
agent overreach (active trop de tâches)
    ↓
attention diluée
    ↓
under-finish (aucune ne passe e2e)
    ↓
complexité du code partiel augmente (mid-state)
    ↓
prochaine session = encore plus dur à reprendre
    ↓
encore plus de tendance à overreach pour "compenser"
```

Briser la boucle = imposer WIP=1.

### Données Anthropic

> "Agents using a 'small next step' strategy (equivalent to WIP=1) show a 37% higher task completion rate than agents using broad prompts." — Anthropic

> "More interestingly, the number of lines of code generated by agents is weakly negatively correlated with actual feature completion — more code written, fewer features completed." — Anthropic research

L'agent qui produit *plus* est généralement l'agent qui *finit moins*.

### Comment encoder WIP=1 dans la harness

#### 1. Dans le `feature_list.json`

```json
{
  "tasks": [
    {"id": "001", "status": "passing"},
    {"id": "002", "status": "active"},
    {"id": "003", "status": "not_started"},
    {"id": "004", "status": "not_started"}
  ]
}
```

Un seul `active` à la fois. La harness peut **bloquer** une transition `not_started → active` si une autre tâche est déjà `active`.

#### 2. Dans les instructions (`CLAUDE.md`)

```markdown
## Work Rules

- WIP=1 : only one task may be `active` at a time.
- Do not start a new task until the current one transitions to `passing`.
- If blocked, set status to `blocked` and write the blocker reason.
```

#### 3. Via hooks (optionnel)

Un PostToolUse hook qui scan le `feature_list.json` et bloque l'écriture si plus d'une tâche est active. Voir [[claude-code-hooks]].

### "Do less but finish" principle

> "Do more but half-done" always loses to "do less but complete". — Lecture 07

Concrètement : 1 feature complètement testée, déployée, documentée bat 5 features "à 80%".

### Lien avec completion evidence

WIP=1 ne suffit pas. Il faut aussi définir clairement quand une tâche transitionne vers `passing` :

- Pas "le code est écrit"
- Mais "la commande de vérification retourne 0" (voir [[completion-evidence-executable]])

Sinon l'agent peut tout marquer `passing` lui-même et redémarrer une autre tâche → on perd le bénéfice.

### Quand augmenter le WIP

Cas rares où WIP > 1 est justifié :

1. **Vraie parallélisation par worktrees** : 2 agents en 2 worktrees, isolés (voir [[git-worktrees-parallel-claude]])
2. **Tâche en blocage externe** (attend un review) : OK de démarrer une autre, marquer la première `blocked`
3. **Long compile / long test** : OK de prototyper une autre pendant le run

Mais : 1 agent + 1 session = WIP=1, sans exception.

### Cas réel (Lecture 07)

Projet : API 8 features. Comparaison :

| Approche | Code total | Features passant E2E | Finales déployables |
|----------|-----------|----------------------|---------------------|
| Unconstrained (WIP=5) | 800 lignes | 20% (1.6/8) | 3/8 |
| WIP=1 | 200 lignes | 100% (8/8) | 7/8 |

**Moins de code, plus de features livrables.**

### Lien avec scope creep

Steve McConnell (*Rapid Development*) identifie le scope creep comme cause #1 d'échec de projet. Pour les agents, le scope creep est **gratuit en tokens** (proposer une idée connexe ne coûte presque rien) → particulièrement dangereux.

WIP=1 = anti-scope-creep mécanique.

### À retenir

1. **WIP=1** par défaut. Une seule tâche active.
2. **Plus de code ≠ plus de features.** Corrélation **négative** observée.
3. Math : `C/k < threshold` → rien ne finit.
4. Vicious cycle overreach→under-finish à briser via WIP=1.
5. Encoder dans `feature_list.json` + dans `CLAUDE.md`.
